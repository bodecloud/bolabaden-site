I think part of the disconnect is that I optimize for reducing friction and cognitive overhead, not for following the “expected” workflow of a system.
When I evaluate software, I’m not just asking “can this be done?” I’m also asking:
how obvious is it?
how many steps does it take?
how much context-switching does it require?
how quickly could a new or distracted user figure it out?
That’s probably influenced by ADHD. If a workflow interrupts momentum, hides common actions, or requires me to mentally model how the developers intended the app to work, it starts feeling inefficient very quickly.

For me, intuitive software minimizes the gap between:
“what I expect to happen”
and
“what the application requires me to learn first.”
That doesn’t make the Linux approach objectively wrong, but it does explain why some workflows feel unnecessarily high-friction to me compared to alternatives.
