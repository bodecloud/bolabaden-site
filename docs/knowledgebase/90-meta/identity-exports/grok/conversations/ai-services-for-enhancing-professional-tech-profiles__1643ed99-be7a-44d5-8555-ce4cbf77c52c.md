## ⚠️ The Only Caveat: IP + Clarity

If the project:

* Is purely built on reverse-engineering (e.g., you wrote a `.gff` parser for KOTOR modding), **that’s awesome**
* But if it includes **assets from the game**, or **copies copyrighted code**, just make sure to:

  * Avoid linking that one directly
  * Move the impressive logic into a standalone lib
  * Make the README about *what you built*, not the game

## ✅ What to Do With Game Projects

### 🛠 1. Pick 1–2 That Show Off Real Skill

Even if they’re game-adjacent, if they:

* Solve a hard problem
* Work well
* Have a clean README

They’re **portfolio-worthy**.

> 📦 [`kotor-gff`](https://github.com/yourrepo) — Python library for parsing and editing `.gff` files used in KOTOR saves and modules. Used by modders to automate builds. No game assets included.

### ✍️ 2. Explain It Like It’s a Technical Challenge

Now it sounds like a **job-ready systems problem**, because it is.

### 💡 3. Mix Game Projects With Utility or Web Projects

To avoid looking one-dimensional, just:

* Pair the game stuff with a **non-game script, web app, or automation**
* Add a **clear summary** of what kinds of jobs you want

That makes it look intentional. Not accidental.
