_Time: 2026-05-29T16:19:25.949Z_

**Yes, GitHub Actions with standard GitHub-hosted runners on personal public repositories is still completely free — and has been for a long time (with very clear official confirmation).** There is **no minute quota or billing** for standard runners (ubuntu-latest, windows-latest, macos-latest, etc.) on public repos.

### Official Current Status (as of May 2026)

GitHub’s documentation states clearly and repeatedly:

> “GitHub Actions usage is **free** for self-hosted runners and for **public repositories** that use **standard GitHub-hosted runners**.”

> “The use of standard GitHub-hosted runners is free: In public repositories.”

This applies to personal accounts on the GitHub Free plan. The 2,000 minutes (Free) or 3,000 minutes (Pro) you see in the billing tables apply **only to private repositories**. Public repository usage of standard runners is tracked separately and is unlimited/free.

**Larger runners** (the bigger CPU/RAM options) are **always charged**, even on public repositories.

**Self-hosted runners**: GitHub does not charge you for the runner minutes themselves (you provide the hardware). There was a planned $0.002/min “platform charge” announced for self-hosted runners in late 2025 (effective March 2026 for private repos), but:
- It explicitly **does not apply to public repositories**.
- The change was later postponed anyway.
- Current docs still list self-hosted usage as free.

So in your case (personal public repo + standard hosted runners), you are in the clear on billing.

### What Limits *Do* Exist?

While minutes are unlimited on public repos, there are still **hard technical/operational limits** that apply to everyone:

| Limit | Details | Relevance to your situation |
|-------|---------|-----------------------------|
| **Job Concurrency** (GitHub Free personal) | 20 concurrent jobs total on standard GitHub-hosted runners (only 5 of which can be macOS) | Most important practical limit. If you push frequently while previous Linux + Windows jobs are still running, new jobs will **queue**. |
| **Matrix jobs per workflow run** | Maximum 256 jobs generated by a single `strategy.matrix` | Unlikely to hit unless you have a huge matrix. |
| **Job timeout (GitHub-hosted)** | 6 hours (hard limit) | Your ~12 min builds are fine. |
| **Workflow run lifetime** | 35 days total (including queued/pending time) | Not relevant for normal CI. |
| **Re-runs** | Max 50 re-runs per workflow run | Fine for testing. |
| **Artifact + Cache storage** | Included amounts exist (500 MB artifacts on Free), but enforcement/billing is different for public repos | Usually not an issue for normal CI artifacts. |

These limits are the same for public and private repos — they’re not billing-related.

### Implications for Your PR / Workflow Testing

- **You can safely approve the workflows.** The “Approve and run” step that appears on PRs from forks (or first-time contributor workflows) is a **security feature**, not a billing/ quota thing. Approving lets the workflows actually execute so you can see if they pass.
- Every push to the PR branch will trigger a new workflow run. With Linux + Windows jobs, that’s roughly two jobs per push. If you push while previous runs are still going, they’ll queue after the 20 concurrent job limit.
- A ~12-minute full build (Linux + Windows) is reasonable and not excessive.
- Because this is a **public repo** using **standard runners**, Modawan’s concern about “exceeding quota” does **not** apply here. The 2,000-minute pool is for private repo usage only.

### Quick Recommendations

1. **Approve the workflows** so you (and the reviewer) can actually see results.
2. If you’re going to do many rapid pushes while testing, consider using the `concurrency` key in your workflows to cancel in-progress runs on the same branch/PR. This keeps the queue cleaner.
3. Double-check you’re not accidentally using any **larger runners** in the workflow YAML.
4. If you ever move these workflows to a **private repo**, then the 2,000-minute monthly limit becomes relevant.

**Bottom line**: For a personal public repository using standard GitHub-hosted runners, Actions is still free and has no meaningful minute-based restrictions. The only things to watch are the normal operational limits (especially the 20 concurrent job cap on the Free plan) and normal CI hygiene (concurrency groups, not spamming pushes, etc.).
