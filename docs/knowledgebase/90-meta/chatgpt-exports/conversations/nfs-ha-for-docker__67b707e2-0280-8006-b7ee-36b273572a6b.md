# NFS HA for Docker

## User

if I use nfs on two different vps hosts, does that mean i have one point of failure, or is the nfs server actually being hosted between both of them? trying to setup docker swarm for high availability with two managers and i'm trying to figure out how to do the filesystems.

## Assistant

If you're using NFS (Network File System) on two different VPS hosts, you typically have a single point of failure unless you've set up a highly available NFS solution.

How NFS Works in Your Case:

NFS is a centralized file-sharing protocol, meaning one host (or cluster) acts as the NFS server and the others act as clients.

If the NFS server goes down, clients lose access to the shared storage.

Your Current Setup:

You have two Docker Swarm managers.

If you’re using NFS and it’s only running on one VPS, that’s a single point of failure (if the NFS host fails, your shared storage is gone).

If both managers are just mounting an NFS share hosted on one of them, it doesn’t provide high availability (HA) for storage.

Solutions for High Availability:

To avoid a single point of failure, you have a few options:

Use a Distributed Filesystem

Consider GlusterFS or Ceph, which provide replication between multiple nodes.

With GlusterFS, for example, both VPS hosts could act as storage nodes, meaning there’s no single NFS server dependency.

NFS High Availability

You can run NFS in an active-passive HA setup with something like DRBD + Pacemaker + Corosync, where:

DRBD replicates the data between nodes.

Pacemaker and Corosync handle failover.

This is more complex but keeps NFS highly available.

GlusterFS vs. Ceph
Feature	GlusterFS	Ceph

Complexity	Easier to set up and manage	More complex, needs more nodes for HA
Replication	Native replication between nodes	More suited for large-scale distributed storage
Performance	Good for small clusters	Better for large-scale deployments
Minimum Nodes for HA	2 (can work well with just two servers)	3+ recommended for proper HA
